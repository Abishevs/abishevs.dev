# Engineering Examples

## Purpose

Source code is often accompanied by execution results.

These two elements represent a single engineering example rather than unrelated code blocks.

The theme should recognize this relationship and render executable examples as unified engineering components.

---

# Engineering Example

An Engineering Example consists of one or more related elements.

Possible structure:

Source

↓

Execution

↓

Output

↓

Generated Figures

↓

Generated Tables

↓

Discussion

Not every example contains every section.

---

# Automatic Detection

The implementation should investigate automatic recognition of executable examples.

Potential indicators include:

* adjacent source and output blocks
* Org Babel exports
* generated figures immediately following source
* generated tables immediately following source

The implementation should remain compatible with standard Org Mode exports.

Avoid introducing mandatory author annotations.

---

# Execution Environments

Presentation should depend on execution environment.

Examples include:

Shell

Python

C

Rust

SQL

Mermaid

PlantUML

Graphviz

Different execution environments naturally communicate different engineering concepts.

---

# Terminal Sessions

Shell-based languages should render using a dedicated Terminal component.

Supported languages include:

* bash
* shell
* sh
* zsh
* fish

The component should resemble a real engineering terminal while remaining consistent with the website design system.

Potential features:

* configurable prompt
* current working directory
* Git branch
* hostname
* user

These values should be configurable globally rather than authored repeatedly.

The prompt should be generated by the renderer rather than stored within content.

---

# Program Output

Program output should be visually distinguished from source code.

Output is not syntax-highlighted code.

It represents execution.

The presentation should communicate this distinction clearly.

Possible approaches include:

* dedicated Output section
* terminal output
* execution panel

The implementation should evaluate which approach provides the clearest reading experience.

---

# Generated Content

Executable examples may produce additional artifacts.

Examples include:

* figures
* tables
* CSV output
* benchmark results
* simulation plots

These should remain visually associated with the originating source block.

Readers should perceive them as part of one engineering example.

---

# Future Metadata

Engineering Examples may eventually expose metadata.

Potential examples:

* execution environment
* filename
* compiler
* interpreter version
* execution time
* memory usage
* platform

The architecture should remain compatible with future metadata without requiring redesign.

---

# Modular Architecture

The implementation should separate:

* source rendering
* execution rendering
* output rendering
* generated artifacts

Each renderer should remain independently maintainable.

---

# Mobile

Engineering Examples should remain readable on small screens.

Avoid layouts that rely on fixed widths or multiple columns.

Terminal sessions should remain horizontally scrollable when necessary while preserving formatting.

---

# Theme Philosophy

The website should present executable engineering examples rather than isolated code blocks.

Readers should immediately understand:

* what was written,
* what was executed,
* what happened,
* what artifacts were produced.

The implementation should emphasize engineering communication rather than Markdown structure.

# Engineering Knowledge System

## Vision

Maker Log is a curated public engineering knowledge system.

It is not a portfolio.

It is not a résumé.

It is not a social media feed.

Instead, it is a long-term engineering notebook documenting projects, ideas, workflows, learning and professional growth.

The website exists to document **how engineering knowledge evolves over time**.

The emphasis is on engineering thinking, experimentation and continuous learning rather than accomplishments.

---

# Core Philosophy

The website should answer different questions through different views of the same underlying knowledge.

Examples include:

* Home → What am I exploring right now?
* Projects → What have I built?
* Timeline → What was I working on during a particular period?
* Work Experience → Where have I applied engineering professionally?
* Education → Where did specific knowledge originate?
* Workflows → How do I work?
* Books → Which ideas influenced my engineering?
* Blog → What have I learned?

No page owns information.

Pages simply present different views over the same underlying knowledge.

---

# Knowledge Nodes

The website consists of independent Knowledge Nodes.

A Knowledge Node represents any meaningful piece of engineering knowledge.

Examples include:

* Projects
* Project Journals
* Articles
* Work Experience
* Education
* Courses
* Books
* Papers
* Workflows
* Technologies
* Tools
* Timeline Events
* Learning Topics
* Future content types

Every Knowledge Node owns its own page.

Knowledge Nodes should be reusable across the website.

---

# Relationships

Relationships are first-class citizens.

Knowledge becomes valuable through context.

Examples:

Project

→ Project Journal

→ Timeline

→ Work Experience

→ Blog Posts

→ Technologies

→ Books

→ Related Projects

Every relationship should add engineering context.

Relationships should be authored intentionally rather than inferred automatically.

Whenever practical, relationships should be bi-directional.

---

# Views

The website is composed of Views.

A View visualizes existing Knowledge Nodes.

Views do not own information.

Examples include:

* Homepage
* Project Index
* Timeline Index
* Search
* Taxonomies
* Future Knowledge Graph

Multiple Views may display the same Knowledge Node without duplication.

---

# Curation

Everything published is intentionally curated.

Nothing should appear automatically simply because it exists.

Publishing is a deliberate engineering decision.

Relationships should communicate meaningful context rather than maximizing connectivity.

Quality of relationships is preferred over quantity.

---

# Engineering Growth

Engineering growth is multidimensional.

Projects evolve alongside:

* Education
* Professional experience
* Reading
* Learning
* Tooling
* Workflows
* Technologies
* Personal experimentation

The website should communicate these parallel evolutions naturally.

Growth should feel continuous rather than segmented into isolated achievements.

---

# Workflows

Engineering workflows are considered engineering knowledge.

Examples include:

* Operating systems
* Desktop environments
* Editors
* Dotfiles
* CAD workflows
* Embedded development environments
* Writing systems
* Automation tools

Workflows are not merely preferences.

They document how engineering methods evolve over time.

---

# Technologies

Technologies represent engineering concepts rather than projects.

Examples include:

* C
* Rust
* Python
* SystemVerilog
* UVM
* KiCad
* FreeRTOS
* STM32

Technology pages should connect projects, articles, work experience and learning.

---

# Books and Resources

Books, papers and other learning resources are part of the engineering knowledge system.

Resources should connect ideas to implementation.

Example:

Book

↓

Inspired Project

↓

Related Article

↓

Timeline Event

↓

Technology

The emphasis is on documenting influence rather than reviewing content.

---

Visitors should naturally move between related knowledge through context.

Example:

Project

↓

Timeline

↓

Work Experience

↓

Technology

↓

Related Project

↓

Blog Post

↓

Book

The website should increasingly resemble an interconnected engineering handbook rather than a collection of isolated pages.

---

# Theme Philosophy

The visual design should support long-form technical reading.

Prioritize:

* Readability
* Context
* Structure
* Information density
* Engineering aesthetics

Avoid:

* Marketing language
* Corporate portfolio layouts
* Startup landing pages
* Decorative interfaces
* Visual clutter

The interface should remain calm, technical and documentation-oriented.

---

# Extensibility

The architecture should assume that new Knowledge Node types will be introduced over time.

The theme should not hardcode assumptions about available content types.

Whenever possible, implementations should favor reusable metadata, layouts and relationship components over content-type-specific solutions.

---

# Future Direction

The website should gradually evolve into an interconnected engineering knowledge graph.

The Knowledge Graph itself is not the objective.

The objective is building a clean, reusable relationship model that can later power:

* Timeline visualizations
* Graph visualizations
* Search
* Backlinks
* Related content
* Future navigation systems

Every new feature should strengthen this shared knowledge model rather than creating isolated functionality.

---

# Guiding Principle

Every page should answer a different question about the same engineering knowledge.

Every feature should strengthen the underlying knowledge model.

Every published node should contribute meaningful context to the larger story of engineering growth.

The website should feel less like a personal homepage and more like a curated engineering archive that visitors can explore from multiple perspectives.
